## 1. Coding Conventions

### TypeScript
- All files must be TypeScript. No `.js` files except config (`next.config.js`, `postcss.config.js`, `tailwind.config.ts`).
- Prefer `interface` over `type` for object shapes. Use `type` for unions and primitives.
- No `any` except in Supabase client return types where the generic is unavailable.
- All database row shapes live in `types/database.ts`. Do not define inline schema types elsewhere.

### Imports
- Use path alias `@/` for all project imports. No relative `../../..` paths except within the same folder.
- Group imports: React/Next → external packages → internal `@/lib` → internal `@/components` → internal `@/types`
- No unused imports. Remove them before committing.

### Naming
- Files: `kebab-case.tsx` for components, `kebab-case.ts` for utilities
- React components: `PascalCase`
- Functions/variables: `camelCase`
- Database tables/columns: `snake_case`
- CSS classes: Tailwind utility classes only — no custom CSS unless using CSS variables in `globals.css`

### Comments
- Write no comments by default.
- Only add a comment when the WHY is non-obvious: a hidden constraint, a workaround, a subtle invariant.
- Never comment WHAT the code does — well-named identifiers do that.

### Error Handling
- At system boundaries (API routes, form submissions): handle errors explicitly, return structured error responses.
- Inside trusted server code: let errors propagate naturally or catch and log without swallowing.
- Activity logging must never throw — it is wrapped in try/catch in `logActivity()`.

## 2. Folder Structure

```
app/
  (public)/          # Public pages — server components by default
  admin/(auth)/      # Login page
  admin/(protected)/ # Admin pages — always call requireAdmin()
  api/admin/         # Admin API route handlers

components/
  admin/             # Admin-only React components
  forms/             # Public-facing form components
  home/              # Homepage section components
  layout/            # Header, footer, breadcrumb, etc.
  packages/          # Package-related components
  sections/          # Section builder components (page-renderer.tsx + 22 section files)
  ui/                # shadcn/ui primitive components (do not modify)

lib/
  activity/          # logActivity() and client variant
  admin/             # guard.ts and api-guard.ts
  homepage/          # Homepage section fetch + registry
  nav/               # Navigation fetch helpers and pool sync
  sections/          # Section metadata (meta.ts), fetch functions (registry.ts), templates
  seo/               # buildMetadata(), fetchSeoContext(), JSON-LD
  supabase/          # createPublicClient, createServerClient, createAdminClient
  theme/             # Theme provider and token helpers
  validation/        # Shared Zod schemas

types/
  database.ts        # Single source of truth for all row types
```

**Rules:**
- Never create a new top-level folder without documenting it in ARCHITECTURE_V3.md.
- Admin-only code never imports from public page components.
- `components/ui/` files are generated by shadcn/ui — do not edit them manually.

## 3. Component Conventions

### Server vs. Client Components
- Default to Server Components. Add `'use client'` only when using hooks (`useState`, `useEffect`, browser APIs).
- Never import server-only modules (`next/headers`, `createServerClient()`, `createAdminClient()`) from client components.
- For shared code used by both client and server: put the client-safe portion in a separate file (e.g. `lib/sections/meta.ts` vs `lib/sections/registry.ts`).

### Page Components
- Each `page.tsx` is a server component.
- Call `createPublicClient()` (not `createServerClient()`) in `generateMetadata` — it runs outside request context.
- Always handle the not-found case: call `notFound()` if the entity does not exist.
- Wrap module detail pages in `<PageRenderer entityType="..." entityId="..." fallback={...} />`.

### Admin Components
- Admin page server components call `requireAdmin()` at the top. If the page fetches data directly, use `createAdminClient()`.
- Admin client components that mutate data call `/api/admin/...` endpoints — never call Supabase directly from the browser.
- Use `toast` (sonner) for success/error feedback. Import from `sonner`, not from `@/components/ui/toast`.

### Forms
- All forms use `react-hook-form` with Zod resolvers.
- Form validation mirrors the API Zod schema — do not duplicate validation logic.
- Submit handlers must disable the submit button during submission (`isSubmitting` from `useFormState`).

## 4. API Conventions

### Route Handler Pattern

Every `/api/admin/*` route follows this pattern:

```typescript
export async function POST(req: NextRequest) {
  // 1. Auth check
  const session = await requireAdminApi();
  if (session instanceof NextResponse) return session;
  const { email } = session;

  // 2. Parse body
  let body: unknown;
  try { body = await req.json(); }
  catch { return NextResponse.json({ error: 'Invalid JSON' }, { status: 400 }); }

  // 3. Validate
  const parsed = schema.safeParse(body);
  if (!parsed.success) {
    return NextResponse.json({ error: 'Validation failed', issues: parsed.error.flatten().fieldErrors }, { status: 422 });
  }

  // 4. Mutate (use createAdminClient())
  const supabase = createAdminClient();
  const { data, error } = await supabase.from('table').insert(parsed.data).select().single();
  if (error) return NextResponse.json({ error: error.message }, { status: 500 });

  // 5. Side effects (nav pool, activity log)
  await logActivity({ action: 'create', entity: 'table', entityId: data.id, userEmail: email });

  // 6. Return
  return NextResponse.json(data, { status: 201 });
}
```

### HTTP Status Codes
- `200` — successful GET
- `201` — successful POST (created)
- `400` — invalid JSON body
- `401` — unauthenticated
- `404` — resource not found
- `422` — Zod validation failure (include `issues` in response)
- `500` — unexpected server error (include `error` message)

### Nav Pool Side Effects
When creating/updating/deleting destinations, vehicle categories, or airport routes, always call the appropriate nav pool helper:
- Create/Update: `await upsertNavPool(...)`
- Delete: `await removeNavPool(entity_type, entity_id)`

## 5. Database Migration Rules

- Every schema change is a new migration file in `supabase/migrations/`.
- Use the `apply_migration` MCP tool — never run `supabase` CLI commands.
- File naming: `YYYYMMDDHHMMSS_NNNN_short_description.sql`
- Never `DROP` columns or tables — they destroy user data.
- Never change a column type if rows exist — add a new column instead.
- Never rename a table — add a view alias if needed.
- Always `ENABLE ROW LEVEL SECURITY` on new tables.
- Write 4 separate RLS policies per table (SELECT, INSERT, UPDATE, DELETE) — never `FOR ALL`.
- Test the migration in isolation before applying — check for column type mismatches.

### Array Column Syntax in SQL
Text arrays (`text[]`) require Postgres array literal syntax:
```sql
-- Correct
INSERT INTO packages (highlights) VALUES ('{"item1","item2","item3"}');

-- Wrong (JSON array syntax)
INSERT INTO packages (highlights) VALUES ('["item1","item2"]');
```

JSONB columns accept JSON syntax:
```sql
INSERT INTO packages (itinerary) VALUES ('[{"day":1,"title":"Arrival"}]'::jsonb);
```

## 6. Git Commit Conventions

Use the format: `type(scope): short description`

**Types:**
- `feat` — new feature
- `fix` — bug fix
- `refactor` — code change without feature or fix
- `docs` — documentation only
- `chore` — build, tooling, dependencies
- `db` — migration or schema change

**Examples:**
```
feat(admin): add page builder section reorder
fix(nav): resolve pool sync on destination delete
db(migration): add hotel_cities table
docs: update ARCHITECTURE_V3 sprint roadmap
chore: upgrade @supabase/ssr to 0.13
```

**Rules:**
- Maximum 72 characters per line in commit message
- No commit merges two unrelated changes
- No commits that break the build (verify with `npm run build` first)

## 7. Pull Request Checklist

Before opening a PR, confirm:

- [ ] `npm run build` passes with zero errors
- [ ] `npm run typecheck` passes with zero errors
- [ ] `npm run lint` passes with zero warnings
- [ ] All new public pages have `generateMetadata`
- [ ] All admin mutations call `requireAdminApi()` and `logActivity()`
- [ ] All new tables have RLS enabled with 4 separate policies
- [ ] `types/database.ts` is updated for any new/changed columns
- [ ] `CHANGELOG.md` entry added
- [ ] `PROJECT_STATUS.md` updated if sprint status changed
- [ ] No `console.log` statements left in production code
- [ ] No hardcoded phone numbers, URLs, or API keys (use `theme_settings` and `site_settings`)
- [ ] Nav pool sync wired for entities that appear in navigation
- [ ] New section types added to `SECTION_TYPE_META` in `lib/sections/meta.ts`

## 8. Code Review Checklist

Reviewers check:

- [ ] Does the change match the architecture described in ARCHITECTURE_V3.md?
- [ ] Is client/server boundary respected? (no `next/headers` imports in client components)
- [ ] Does the API route validate input with Zod?
- [ ] Is the Supabase client choice correct? (public for reads, admin for mutations)
- [ ] Are there any SQL injection risks? (Supabase query builder is safe; raw SQL must be parameterised)
- [ ] Does the component have correct TypeScript types (no implicit `any`)?
- [ ] Are error and empty states handled?
- [ ] Does the page render correctly at mobile viewport (320px+)?

## 9. Build Verification Checklist

Run before every deployment:

```bash
npm run build
npm run typecheck
npm run lint
```

All three must pass with zero errors and zero warnings.

After applying any migration:
- Verify row counts in affected tables with `execute_sql`
- Confirm RLS policies exist (`SELECT * FROM pg_policies WHERE tablename = 'your_table'`)
- Test the affected admin CRUD routes manually
