Document 01
Product Requirement Document (PRD)
1.1 Application Purpose

Divine Travel is a full-stack travel agency web platform serving two distinct audiences: public visitors who discover pilgrimage, domestic, and international tour packages, submit inquiries, read travel blogs, and view testimonials; and admin staff who manage all content, handle leads through a CRM-style sales pipeline, and configure site branding, SEO, and homepage layout.

1.2 Target Users
PersonaDescription
Prospective TravelerIndividual or family seeking pilgrimage / leisure tours from India
Pilgrim / DevoteeUser specifically looking for divine/spiritual yatra packages
Honeymoon CoupleCouple seeking romantic domestic/international packages
Admin / Sales AgentInternal staff managing leads and content via the admin dashboard
Marketing ManagerStaff configuring SEO, analytics IDs, and homepage layout
1.3 Business Goals
#Goal
1Generate qualified leads through inquiry forms (contact, quick quote, package inquiry)
2Showcase tour packages to drive booking enquiries
3Manage the full lead lifecycle from initial contact to booking confirmation
4Maintain SEO-optimized content (packages, blogs, destination pages) to drive organic traffic
5Provide admin team a centralized CMS for all site content
6Track marketing attribution via UTM parameters stored on every lead
1.4 Core Modules
ModuleScope
Homepage BuilderDynamic section management — enable/disable and reorder 10 homepage sections
Package CatalogCRUD, public listing with search/filter/sort, detail pages with structured data
Lead ManagementCapture via forms, Kanban pipeline (6 statuses), notes, priority, UTM attribution
Blog CMSCreate/edit/publish blog posts with auto reading time and SEO fields
Destination ManagementRegion-specific destination pages (divine / domestic / international)

SEO ManagementPer-page SEO overrides, robots, canonical, JSON-LD injection

Document 02
Software Requirement Specification (SRS)
2.1 Lead Capture Form
Feature Name
Lead Capture Form
Description
Collects visitor contact and travel intent details and stores as a lead in the CRM pipeline
Inputs
name (required), mobile (required, Indian regex), email (optional), destination, travel_date, adults, children, budget, message, source (enum), package_id, package_slug
Outputs
Lead record in DB; activity log entry; JSON response { ok: true, id, message }
Business Rules
Mobile must match Indian phone regex; travel_date must be future date; rate limit 5 requests / 60 s / IP; UTM params parsed from cookies; source distinguishes origin (contact / quick-quote / package-inquiry / callback)
Dependencies
Supabase leads table; /api/leads route; Zod leadSchema
2.2 Package Catalog
Feature Name
Package Catalog
Description
Browseable, searchable, filterable listing of all published tour packages
Inputs
Search query; category filter; sort selection (updated / price_low / price_high / duration)
Outputs
Filtered package cards: title, price, duration, cover image, category badge
Business Rules
Only is_published = true packages shown publicly; hierarchical categories (parent/child); is_featured flag controls homepage section
Dependencies
Supabase packages, package_categories tables
2.3 Package Detail Page
Feature Name
Package Detail
Description
Full single-package view with all content sections and inquiry CTA
Inputs
URL slug
Outputs
Overview, itinerary (day-by-day), pricing, inclusions/exclusions, gallery, FAQs, related packages, inquiry form, sticky CTA, JSON-LD (TouristTrip + BreadcrumbList + FAQPage)
Business Rules
Must return 404 for unpublished or non-existent slugs; related packages from same category; structured data mandatory in page source
Dependencies
packages, package_categories tables; SEO metadata system
2.4 Blog System
Feature Name
Blog
Description
Travel content articles with category filtering and full Markdown rendering
Inputs
Search query; category filter
Outputs
Blog cards (list); full article with ToC, author, publish date, reading time, related posts; JSON-LD Article
Business Rules
Only is_published = true shown publicly; reading_time_minutes auto-calculated (words / 200, min 1); published_at auto-set on first publish
Dependencies
blogs table; Markdown renderer component
2.5 Lead Pipeline (Admin)
Feature Name
Lead Pipeline
Description
CRM-style Kanban board for managing leads from initial contact to confirmed booking
Inputs
Drag-drop between columns; status filter; search (name / mobile / destination)
Outputs
Lead cards with priority, assigned user, follow-up date; status auto-updated with audit note and timestamp
Business Rules
6 statuses: new → contacted → quoted → negotiation → confirmed → lost; 4 priority levels: low / medium / high / urgent; max 500 leads fetched; status changes appended to notes JSONB array with timestamp
Dependencies
leads table; activity log

FeatureKey Business RulesDependencies

Document 05
Use Cases
UC-001 — Submit Lead via Contact Form
Actor
Public visitor
Preconditions
Visitor is on /contact page
Main Flow
1. Visitor fills Name and Mobile (required)
2. Optionally fills Email, Destination, Travel Date, Adults, Children, Budget, Message
3. Clicks "Send Enquiry"
4. Client-side Zod validation passes
5. POST to /api/leads
6. Rate limit check passes
7. Lead created in DB with source="contact"
8. Activity logged; UTM cookies parsed and stored
9. Success message replaces form
Alternate Flow
Visitor fills only Name and Mobile → lead still created successfully with minimal data
Exceptions
E1: Validation fails (mobile format wrong, past travel date) → inline error, form not submitted
E2: Rate limit exceeded (>5 in 60s) → HTTP 429, error toast
E3: DB insert fails → HTTP 500, error toast
UC-002 — Browse and Filter Packages
Actor
Public visitor
Preconditions
Published packages exist in DB
Main Flow
1. Visitor navigates to /packages
2. All published packages displayed
3. Visitor types in search box → list filters in real time
4. Visitor clicks a category in sidebar → list filters
5. Visitor changes sort dropdown → list reorders
Alternate Flow
Visitor navigates with category query param → category pre-selected on load
Exceptions
E1: No packages match filter → "No packages found" empty state
E2: No published packages at all → empty state on initial load
UC-003 — Admin Moves Lead Through Pipeline
Actor
Admin / Sales Agent
Preconditions
Admin authenticated; leads exist in DB
Main Flow
1. Admin navigates to /admin/leads
2. Kanban board displayed with 6 status columns
3. Admin drags a lead card from "New" to "Contacted"
4. System updates lead status in DB
5. Auto-note appended with timestamp
6. Column counts update
Alternate Flow
Admin switches to List view → opens lead detail drawer → changes status via dropdown
Exceptions
E1: Drag fails due to network error → optimistic update reverted, error toast
UC-004 — Create and Publish a Package
Actor
Admin
Preconditions
Admin authenticated; package categories exist
Main Flow
1. Admin navigates to /admin/packages → clicks "New Package"
2. Fills required fields: slug, title, duration, overview
3. Adds itinerary, pricing, inclusions/exclusions
4. Sets is_published = true; optionally is_featured = true
5. Clicks Save → POST to /api/admin/packages
6. Package created; admin redirected to package list
Alternate Flow
Admin saves with is_published = false → package exists in DB but not visible publicly
Exceptions
E1: Slug already exists → unique constraint error returned
E2: Required field missing → Zod validation error shown on form field
UC-005 — Admin Configures Homepage Layout
Actor
Admin / Marketing Manager
Preconditions
Admin authenticated; homepage_sections rows exist
Main Flow
1. Admin navigates to /admin/homepage-builder
2. Sees 10 sections with enable/disable toggles and order numbers
3. Toggles a section off; changes display order of another
4. Saves → homepage_sections table updated
5. Public homepage reflects changes on next page load
Alternate Flow
Admin edits section config JSON (e.g. hero title text) → config JSONB updated, homepage renders with new text
Exceptions
E1: All sections disabled → homepage renders as empty page (no sections)
UC-006 — Admin Login
Actor
Admin
Preconditions
Admin account exists in Supabase Auth
Main Flow
1. User navigates to /admin/login
2. Enters email and password → clicks Sign In
3. Supabase Auth validates credentials
4. Session cookie set; middleware detects session
5. Redirected to /admin dashboard
Alternate Flow
Already authenticated user navigates to /admin/login → middleware redirects to /admin
Exceptions
E1: Wrong credentials → error message shown, no session created
E2: Unauthenticated user navigates to /admin/* → redirect to /admin/login
